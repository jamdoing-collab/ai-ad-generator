## 架构优化（延后处理）

### 1. database.js 职责拆分（等 800 行或新增加解密场景时做）
- **现状**：608 行，同时承担数据库初始化、建表迁移、CRUD、AES 加解密、防抖持久化
- **需改**：AES 加解密逻辑抽成 `server/utils/crypto.js`，database.js 只管数据操作
- **不急的原因**：当前体量可维护，拆分需改所有 require 路径

### 2. sql.js 内存架构替换（等日活千级时做）
- **现状**：整个 DB 运行在内存，定时序列化到磁盘，kill -9 或 OOM 最多丢 2 秒数据
- **需改**：换 `better-sqlite3`（原生 SQLite 直接写磁盘），重写 database.js IO 层
- **不急的原因**：低并发下 2 秒丢失几乎不会发生；不是加备份脚本能解决的，根因是架构

### 3. 参考图公开可访问
- **现状**：grsai API 的参考图需要通过 URL 传入，本地服务器的图片无法被 grsai 访问
- **需改**：部署到公网服务器，或使用 OSS 等对象存储中转参考图
- **不急的原因**：生图（无参考图）功能不受影响
